A skill is a recipe. LLM-as-judge is the technique. The tiger parent is a recipe that tells Claude exactly how to judge your work.

The Key Insight

You don't need to be an expert
to catch problems — you just need
a good evaluator.

1. You ask Claude to make a plan
2. You're not sure if it's good
3. Tiger parent evaluates it: "No rate limiting. No tests. 68/100."
4. Now you know what to ask for next

It turns "I don't know what I don't know" into a concrete punch list.

Why It Works

Different instructions,
different questions

The Planner

"What did the user ask for?
Let me address that."

Your prompt says "build me an API."
The plan builds you an API.

The Evaluator

"Is there error handling? Security?
Edge cases? Testing? Performance?"

The tiger parent says "where's the rate limiting, the auth, the error handling?"

The gap between those two is where the blind spots live.

Architecture

Skill vs sub-agent

Skill

Giving Claude a new hat

  • Same conversation, same context
  • Claude reads new instructions inline
  • Conversational — you can push back
  • Simpler, less overhead
Sub-Agent

Hiring a contractor

  • Separate Claude instance, own context
  • Only sees the output, not the reasoning
  • Returns a verdict — no negotiation
  • Can run multiple in parallel

"A sub-agent sounds fancy. It's just: run the same LLM twice with different prompts."

The Architecture Is Cultural

The builder doesn't get to
argue back

"If you grew up with a tiger parent, you know — you don't talk back. You take the feedback, you fix it, and you come back with a better version."

That's literally what this eval does. Claude submits its plan, the tiger parent tears it apart, and Claude doesn't get to say "but I thought—". It just goes back and does better.

The power dynamic is one-directional by design.

Grading Scale

Every grade is a disappointment

Score Label Translation
100 Adequate "The neighbour's agent would not have made these mistakes."
96–99 Below Expectations "This is what you show me after all the compute I spent on you?"
< 95 Unacceptable "What did I do in my past life to deserve an agent like you?"

"A 100 is theoretically possible. You've never seen one."

Output Format

Five sections. Every time.

1. The Grade

0–100 with justification

2. What's Right

Brief. This is a receipt,
not a celebration.

3. Disappointments

Critical → Significant → Minor

4. What's Missing

"Why didn't you also…"

5. The Verdict

Usable? Needs revision?
Full redo?

Tone

The key calibration

"The comedy comes from the contrast between genuine quality and the impossibly high bar of the reaction. A solid B+ getting treated like a crisis is funny. Actual bad work getting roasted is just mean."
  • Every criticism must be substantive and actionable
  • Never cruel about things an agent can't control
  • Tough love is still love
  • The disappointment is universal — every culture has a version of it
The Cultural Layer

"Log kya kahenge?"

(What will people say?)

"Did you consider how other developers will read this code? What about the next team that maintains it?"

You get a 95, they ask about the other 5. Your plan handles auth, error handling, and testing?
"Where's the rate limiting? The caching strategy? Why didn't you consider observability?"

You will never get a perfect score. And that's the point — because in production, nothing is ever perfect either.
